# T8.b — Non-Apex Extraction-Coverage Audit (Design Stub)

> **Status.** Stub — postponement receipt. No implementation is
> scheduled. T8 shipped R39 + R41 for Apex only; the non-Apex
> coverage story is open.
> **Parent.** [`T8-coverage-awareness.md §8`](T8-coverage-awareness.md)
> (deferred out-of-scope follow-up).
> **Consumed by.** [`../CUSTOMER_READINESS_GATE.md §4.2`](../CUSTOMER_READINESS_GATE.md)
> — non-Apex coverage blocks Bar-C on any language besides Apex.

This doc does *not* author resolvers or extractor queries.
It inventories what we already know is unwalked, per language,
so that when T8.b is scheduled the audit does not start from
zero. Every item below is a candidate `CoverageGap::*` variant
— whether it *is* a variant in a future T8.b PR depends on
whether the downgrade predicate (`invalidates_no_callers` /
`invalidates_fan_in_metrics` / `invalidates_coupling`) can be
defended for that shape.

Non-goals of this stub:

- Not picking which language ships first. That is a
  product / customer-signal decision, not an engine decision.
- Not proposing implementation effort estimates. Each
  per-language audit is its own cost; the audit is the first
  step of the ticket, not its precondition.
- Not claiming the lists below are complete. They reflect the
  shapes we already noticed during T8 review. A full audit
  will surface more.

---

## 1. Framing

T8 established a contract:

1. The extractor emits a `FileExtractionCoverage` row per
   parsed file, carrying a `coverage_gaps: Vec<CoverageGap>`.
2. Each `CoverageGap` variant declares one of three
   invalidation predicates (`invalidates_no_callers`,
   `invalidates_fan_in_metrics`, `invalidates_coupling`).
3. `graphengine-analysis::health::coverage_attach` then
   downgrades the relevant metric's confidence from
   `High → Medium` for any node whose file carries an
   invalidating gap.

T8.b extends that contract to non-Apex languages. The
extractor's walk-completeness for each language is currently
unaudited; the `HealthReport` therefore presents
`no_callers_high_confidence` as if the extractor walks every
shape, which we have no evidence for outside Apex. That is
the measured-fidelity gap this ticket closes.

For every language below:

- Each section lists **confirmed unwalked shapes** (seen
  during tree-sitter grammar review or hand-audit) and
  **suspected shapes** (logical candidates that have not
  been verified against real repos).
- A confirmed shape means we have a fixture or a grep result
  showing the extractor's query set does not match it. A
  suspected shape means we have a plausible reason to doubt
  coverage but no fixture yet.
- The pattern template uses `CoverageGap::<LangSlug><Shape> { count: u32 }` —
  same naming convention as
  [`CoverageGap::ApexPropertyAccessor` / `ApexMapLiteralInitializer`](
  ../../../graphengine-parsing/src/application/ports.rs).

## 2. Java

Baseline: `graphengine-parsing/configs/java.yaml` carries the
Tree-sitter query set. The extractor is
`graphengine-parsing/src/syntax/extractors/*.rs` generic paths;
Java has no language-specific `syntax/language/java/` subtree.

### 2.1 Confirmed unwalked shapes (from review)

None currently. This section is empty because no Java hand-audit
has been run at T8's quality bar. That is itself a finding:
Java's fidelity claim is untested at this level of scrutiny.

### 2.2 Suspected shapes (plausible candidates for audit)

Each needs a fixture + extractor walk to confirm.

- `record` component accessors. Implicit getters generated by
  `public record Foo(int bar)`. The tree-sitter grammar emits
  a single `record_declaration`; whether the extractor walks
  each component into a method symbol (for the accessor) is
  unaudited.
  **Candidate variant.** `CoverageGap::JavaRecordAccessor { count: u32 }`.
  **Reason it could matter.** `no_callers` — any `foo.bar()`
  call site would be unresolved and drop from the graph.
- Lambda + method-reference callees. `Stream.of(1,2,3).map(Foo::bar)`
  is a call site whose callee is `bar`. Tree-sitter parses
  method references as `method_reference` nodes; the generic
  extractor reference-walker may or may not produce a call-site
  for them.
  **Candidate variant.** `CoverageGap::JavaMethodReferenceTarget { count: u32 }`.
  **Reason.** `no_callers` + `fan_in_metrics`.
- Annotation-processed sources (`@AutoValue`, Lombok
  `@Data`, Mockito `@InjectMocks`). These generate callers /
  callees that don't appear in the source tree-sitter walks.
  Apex has an analogous story with `@InvocableMethod` /
  `@AuraEnabled`; the Java side has no equivalent registry.
  **Candidate variant.** `CoverageGap::JavaAnnotationGenerated { count: u32 }`
  (probably a union variant per annotation family, not a single
  count).
  **Reason.** `no_callers` and `framework_annotation_unresolved`.
- Interface `default` methods + `sealed` class hierarchies.
  Both are tree-sitter-walkable but the trait-dispatch
  edge-kind is currently not emitted (the `Extends` / `Implements`
  pair is, but the default-method callee override is not).
  **Candidate variant.** `CoverageGap::JavaDefaultMethodDispatch { count: u32 }`.
  **Reason.** `no_callers` for any call that lands on a
  default-method implementation.
- Inner-class + anonymous-class callee resolution. `new Runnable() { public void run() { ... } }`.
  The tree-sitter grammar exposes the body, but the FQN for
  the anonymous callee is extractor-implementation-dependent.
  **Candidate variant.** `CoverageGap::JavaAnonymousInnerClassCallee { count: u32 }`.
  **Reason.** `fan_in_metrics`.

### 2.3 First audit step for Java

Run `ge-analyze` against `commons-lang`, extract the
`FileExtractionCoverage` records (always `None` today,
because the Java path doesn't emit any), and then hand-diff
a sample of 50 source files against the generated call-edge
set. Any call site that visibly exists in source but does
not appear as a call edge becomes a candidate shape.

## 3. Rust

Baseline: `graphengine-parsing/configs/rust.yaml` plus the
language-specific `graphengine-parsing/src/syntax/rust/*`
(T6-landed). Layer 2 (`RustLayer2SemanticResolver`) already
provides Authoritative-tier resolution for the call shapes
it sees.

### 3.1 Confirmed unwalked shapes

- **Proc-macro expansions.** Tracked under
  [`../FOLLOWUPS.md` UF-FU-003](../FOLLOWUPS.md). The
  adapter reports `FileNotInProjectModel` + `NoTarget`
  misses; 88.9 % of call references miss the adapter (see
  [`INCEPTION_REPORT.md §4`](../INCEPTION_REPORT.md)). This
  is already a measurement, not a surprise — but T8.b's
  job is to produce a `CoverageGap::RustProcMacroExpansion`
  record so the `HealthReport` carries this as a
  `no_callers` downgrade instead of only as a Layer-2 miss
  counter.
  **Candidate variant.** `CoverageGap::RustProcMacroExpansion { count: u32 }`.
  **Reason.** `no_callers` — a function referenced only
  from a proc-macro-expanded call site will never show
  an edge, so `no_callers` is necessarily noisy for that file.

### 3.2 Suspected shapes

- `macro_rules!` bodies. Tree-sitter walks them as token
  trees, but the extractor does not interpret the macro
  body. Any function referenced only from inside a
  `macro_rules!` expansion site is invisible.
  **Candidate variant.** `CoverageGap::RustMacroRulesBody { count: u32 }`.
  **Reason.** `no_callers`.
- `#[cfg(...)]`-gated code. The extractor walks all
  branches today (honest behaviour), but the Layer-2
  adapter only sees the build target's active cfg set.
  Mismatch can produce `High`-confidence results on
  inactive branches (which we should not produce).
  **Candidate variant.** `CoverageGap::RustCfgBranchMismatch { count: u32 }`.
  **Reason.** `no_callers_high_confidence` (specifically
  the dual-metric column).
- Trait-object dispatch (`dyn Trait`). `ra_ap_ide` resolves
  to the trait method declaration, not the concrete impl.
  Whether that counts as a gap or a Medium-confidence
  correct answer is an open call the audit has to make.
  **Candidate variant.** `CoverageGap::RustTraitObjectDispatch { count: u32 }`
  (debatable — may instead be a per-edge confidence clamp).
  **Reason.** `fan_in_metrics`.

### 3.3 First audit step for Rust

UF-FU-012 item (c) (adapter-error histogram) is a
prerequisite. Once landed, classify each per-variant error
to either a concrete extractor-side gap (→ `CoverageGap::Rust*`)
or a Layer-2 adapter limit (→ not a gap, a Medium-confidence
fact).

## 4. TypeScript / JavaScript

Baseline: `graphengine-parsing/configs/typescript.yaml` +
`javascript.yaml`. Both share the generic extractor; no
language-specific subtree. The TypeScript LSP adapter is
in-flight (WS-PROOF-R3 + the TypeScript template-query
contract per the pre-sprint commits) but not at Authoritative
tier.

### 4.1 Suspected shapes

- **JSX / TSX component usage.** `<Foo prop={bar} />`
  desugars to `Foo({prop: bar})`. The extractor's call-site
  walk likely does not treat JSX elements as call sites
  today, so every component usage is invisible to `fan_in`.
  **Candidate variant.** `CoverageGap::TypeScriptJsxElementCallee { count: u32 }`.
  **Reason.** `no_callers` + `fan_in_metrics`.
- Object spread / destructuring callees.
  `const { fn: myFn } = mod; myFn();` — the tree-sitter
  grammar parses the rename, but extractor FQN-building
  may not.
  **Candidate variant.** `CoverageGap::TypeScriptDestructuredCallee { count: u32 }`.
  **Reason.** `no_callers`.
- Decorators (`@Injectable`, `@Controller`, NestJS et al).
  Same annotation-processing story as Java.
  **Candidate variant.** `CoverageGap::TypeScriptDecoratorAnnotated { count: u32 }`.
  **Reason.** `framework_annotation_unresolved`.
- Dynamic `import()` callees. The tree-sitter grammar yields
  a `call_expression` whose callee is the string literal;
  whether we follow it into a module resolution is unaudited.
  **Candidate variant.** `CoverageGap::TypeScriptDynamicImport { count: u32 }`.
  **Reason.** `coupling`.

### 4.2 First audit step for TypeScript

Pick a canary repo with both `.ts` and `.tsx` and run
`ge-analyze`. Audit a sample of React component files
for JSX-originated call sites not in the edge set.

## 5. Python

Baseline: `graphengine-parsing/configs/python.yaml`. Generic
extractor. No Layer-2 adapter today.

### 5.1 Suspected shapes

- Decorator-registered callers (Flask `@app.route`, FastAPI
  `@router.get`, Celery `@shared_task`). Same story as Java
  annotations + TypeScript decorators.
  **Candidate variant.** `CoverageGap::PythonDecoratorRegistered { count: u32 }`.
  **Reason.** `framework_annotation_unresolved` + `no_callers`.
- Dunder methods (`__call__`, `__getattr__`, `__init__`).
  The extractor may walk the defs but not produce call-site
  edges for `obj()` → `obj.__call__()`.
  **Candidate variant.** `CoverageGap::PythonDunderDispatch { count: u32 }`.
  **Reason.** `no_callers` + `fan_in_metrics`.
- Dynamic attribute access (`getattr(obj, "method_name")`).
  By construction unresolvable without runtime, but the
  gap count tells the consumer "this file has `N`
  unresolvable-by-design call sites".
  **Candidate variant.** `CoverageGap::PythonDynamicAttrAccess { count: u32 }`.
  **Reason.** `no_callers` — not a bug, a measured limit.
- Meta-classes / ABC registrations.
  **Candidate variant.** `CoverageGap::PythonMetaclassRegistered { count: u32 }`.
  **Reason.** `framework_annotation_unresolved`.

### 5.2 First audit step for Python

Pick a canary: a Flask app (for decorator-heavy),
a Django monolith (for meta-class heavy), or a scientific
Python codebase (for dunder + dynamic). The choice is
product-signal dependent.

## 6. C#

Baseline: `graphengine-parsing/configs/csharp.yaml`. Generic
extractor. No Layer-2 adapter.

### 6.1 Suspected shapes

- Auto-property accessors (`public int Foo { get; set; }`).
  Same shape as the Apex R39 property-accessor issue.
  **Candidate variant.** `CoverageGap::CSharpAutoProperty { count: u32 }`.
  **Reason.** `no_callers`.
- Partial classes. Tree-sitter walks each file in isolation;
  the FQN builder may not stitch the two halves.
  **Candidate variant.** `CoverageGap::CSharpPartialClassSplit { count: u32 }`.
  **Reason.** `fan_in_metrics` + `coupling`.
- LINQ query-syntax callees (`from x in y select z`).
  The compiler rewrites to `Select`, `Where`, etc. The
  tree-sitter walk should see the method-syntax version
  but not the query-syntax version.
  **Candidate variant.** `CoverageGap::CSharpLinqQuerySyntax { count: u32 }`.
  **Reason.** `no_callers`.
- Async / await state machines. The `await foo()` call site
  is real, but compiler-generated state-machine callees are
  not.
  **Candidate variant.** `CoverageGap::CSharpAsyncStateMachine { count: u32 }`.
  **Reason.** typically no direct metric invalidation, but
  worth noting for narrative completeness.
- Source generators (e.g. ASP.NET Core minimal APIs). Same
  annotation-processing story as Java / TypeScript.
  **Candidate variant.** `CoverageGap::CSharpSourceGenerated { count: u32 }`.
  **Reason.** `framework_annotation_unresolved`.

### 6.2 First audit step for C#

Pick a canary: an open-source `.NET` library (e.g. NodaTime,
already-used as a clean small-library example) or a Razor /
Blazor project. The Razor case may reveal additional
template-engine coverage gaps.

## 8. VisualForce / LWC / Aura (non-language but relevant)

Baseline: `graphengine-parsing/configs/visualforce.yaml`.
Apex's declarative-wiring neighbours. The
[`PHASE_4_DECISION_MEMO.md`](../PHASE_4_DECISION_MEMO.md)
Option 1 scope lands their resolvers — which is where
the `CoverageGap` records would naturally follow.

### 8.1 Confirmed unwalked shapes (from review)

- LWC component `<lightning-*>` references into Apex
  `@AuraEnabled` methods.
- Aura controller `.js` files calling Apex via
  `$A.enqueueAction`.
- VisualForce page `{!action}` bindings.

Candidate variants would live in the Apex neighbourhood
namespace (`CoverageGap::ApexLwcBinding`, etc.) per the
naming convention above.

## 9. Audit sequencing notes

When T8.b is ticketed:

1. Pick one language per round.
2. Stand up a canary fixture per round (or reuse the
   `experiments/results/` canary infrastructure already
   built for NPSP).
3. Hand-diff a sample of files to classify each unwalked
   shape as "confirmed gap → new `CoverageGap` variant" or
   "edge-kind clamp → no gap needed".
4. For each confirmed gap, add: (a) the variant, (b) the
   extractor's counter, (c) a regression test, (d) the
   `invalidates_*` predicate, (e) a NPSP-style canary
   measurement showing the downgrade effect.
5. Append the canary retrospective to
   [`T8-coverage-awareness.md`](T8-coverage-awareness.md)
   §9 (keep one retrospective table per language).

This is the mechanical version of what T8 did for Apex.
Each language-round should feel like repeating T8's
2026-04-21 process without re-deriving the machinery.
