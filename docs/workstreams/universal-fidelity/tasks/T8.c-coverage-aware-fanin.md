This doc does *not* pick a schema or a downgrade rule.
It inventories the machinery T8 already landed and names the
three decisions T8.c has to make.

---

## 1. Current state (as of T8 landing)

`graphengine-parsing/src/application/ports.rs:304`
declares the predicate:

```rust
pub fn invalidates_fan_in_metrics(&self) -> bool {
    self.count() > 0 && matches!(self, /* variant list */)
}
```

…and the comment on that predicate states that fan-in is
"continuous, not boolean — even a single missed edge
perturbs the metric" (see full text at that line in the
authoritative source).

But no consumer of that predicate exists today.
[`graphengine-analysis/src/health/coverage_attach.rs`](../../../graphengine-analysis/src/health/coverage_attach.rs)
only wires `invalidates_no_callers`, and the
`HealthReport` fan-in surface
([`graphengine-analysis/src/health/report.rs`](../../../graphengine-analysis/src/health/report.rs))
carries `fan_in: u32` as a plain count. There is no
`fan_in_high_confidence` column, no per-node `fan_in_confidence`
field, and no coverage-aware downgrade path.

So:

- The *predicate* is declared.
- The *downgrade pipe* is not built.
- The *dual-metric contract* established for `no_callers`
  (`no_callers_total` vs `no_callers_high_confidence`) has
  no fan-in sibling.

This is a deliberate choice from T8 — ship one metric's
end-to-end pipe honestly, not two metrics halfway. T8.c
picks up the second.

## 2. The three open decisions

### 2.1 Node-level vs file-level downgrade granularity

`no_callers` is naturally a node-level boolean ("does this
function have any callers?"). `fan_in` is a node-level
integer. When `file F` has a gap of shape
`CoverageGap::ApexPropertyAccessor { count: 3 }`,
what does that say about fan-in *from* F's functions?

Two candidate rules:

1. **File-coarse rule.** Every node defined in F has its
   fan-in clamped to Medium confidence. Simple, obviously
   correct, over-sensitive (many of F's nodes have no
   relationship to the unwalked shape).
2. **Node-fine rule.** Only nodes whose fan-in *could*
   mechanically be affected by the unwalked shape get
   clamped. For `ApexPropertyAccessor`: only the property
   field and its enclosing class. For
   `ApexMapLiteralInitializer`: only functions referenced
   inside map-literal initializers (which we cannot know
   without walking them, by construction — so fall back
   to the whole class scope).

**Recommendation for T8.c v1.** File-coarse. Matches the
`no_callers` downgrade's granularity and avoids pretending
we know per-node which fan-in shifts. Revisit in a v2 if
Desktop complains about over-downgrading.

### 2.2 Dual-metric schema shape

The `no_callers` split landed as two `Option<u32>` fields
on `DeadCodeMetricDetail`:

- `no_callers_total`
- `no_callers_high_confidence`

For fan-in, two candidate shapes:

1. **Per-node split field.** Each node carries
   `fan_in_total: u32` and `fan_in_high_confidence: u32`.
   Simple, matches the existing per-node `fan_in` field's
   location. Wire cost is a schema migration on the report
   JSON.
2. **Per-node confidence enum.** Each node carries
   `fan_in: u32` and `fan_in_confidence: Confidence` (the
   same `High/Medium/Low` enum already on `dead_code_confidence`).
   Cheaper wire cost (one enum instead of a second int),
   but loses the count-with-gaps-vs-count-without-gaps
   distinction that the `no_callers` split gives.

**Recommendation for T8.c v1.** Option (1) — mirror the
`no_callers` split exactly. Desktop's T3 trend panel
(WS-DESKTOP-C.6) is easier to render against two numbers
per node than one-number-plus-enum.

### 2.3 Scope of the predicate list

Today `invalidates_fan_in_metrics` fires for any
CoverageGap that `invalidates_no_callers` fires for — the
variant lists are identical. That's conservative. T8.c v1
should audit the list per-variant: `ApexMapLiteralInitializer`
definitely affects fan-in (the gap hides call edges);
`ApexPropertyAccessor` less so (the unwalked accessor
bodies hide local references to the property field, not
outgoing call edges, so the effect is on the *targets'*
fan-in rather than the property-holder's).

**Recommendation for T8.c v1.** Leave the lists identical.
The false-positive cost of over-downgrading fan-in is
measurable and recoverable; the false-negative cost of
missing a real gap is product-trust. Audit in a v2 if a
canary measurement proves the downgrade rate unacceptably
high.

## 3. Consumer dependency — WS-DESKTOP-C.6

The Desktop trend view compares two `HealthReport`s and
surfaces per-node metric deltas. Without T8.c:

- A node whose fan-in moved from 12 → 7 between scans
  cannot be distinguished from a node whose fan-in stayed
  at 12 but the second scan's extractor hit a coverage
  gap that hid 5 edges.
- WS-TRUTH-R35's byte-identical determinism gate helps
  only for same-SHA repeats; it does not help for
  cross-commit trend analysis, which is precisely what
  WS-DESKTOP-C.6 is for.

So T8.c is on the critical path for WS-DESKTOP-C.6 to
produce trend judgements customers can trust. Until T8.c
lands, Desktop should either:

- (a) Show fan-in trends only with an "unverified" watermark
  until the consumer reads T8.c's dual-metric output, or
- (b) Suppress fan-in deltas entirely and surface only
  `no_callers` deltas (which do carry the confidence split).

The choice is Desktop's, not the engine's. This stub is
the postponement receipt the engine owes Desktop.

## 4. Implementation sketch (non-binding)

When T8.c is ticketed, the likely PR shape:

1. Extend `DeadCodeMetricDetail` (or sibling struct) with
   `fan_in_total: Option<u32>` and `fan_in_high_confidence: Option<u32>`
   fields, defaulted to `None`.
2. Extend `NodeAnnotation` with `fan_in_confidence: Option<Confidence>`.
3. Add `apply_extraction_coverage_downgrade_to_fan_in` in
   [`coverage_attach.rs`](../../../graphengine-analysis/src/health/coverage_attach.rs),
   symmetric to the existing `no_callers` downgrade.
4. Add `recompute_fan_in_confidence_split` called from
   the `ge_analyze` binary after the existing no-callers
   split.
5. Persisted-schema migration in
   `graphengine-parsing/src/infrastructure/storage/schema.rs`
   is **not** required — the fan-in count itself is already
   derived from the edge table at analysis time, and the
   `FileExtractionCoverage` table already persists the
   gaps. T8.c is purely a consumer-side change.
6. Integration test at `graphengine-analysis/tests/t8c_*.rs`
   mirroring the `t8_dead_code_respects_extraction_coverage.rs`
   shape:
   - File with no gaps → fan-in stays `High`, counts equal.
   - File with an `invalidates_fan_in_metrics` gap → both
     counts present, high-confidence count ≤ total.
   - Zero-count gap → no downgrade (same bug fixed in T8
     for `no_callers`).
7. NPSP canary retrospective appended to
   [`T8-coverage-awareness.md`](T8-coverage-awareness.md)
   §9 as the Apex-specific fan-in measurement.

## 5. Out-of-scope for T8.c v1

- **Cycles metric.** `invalidates_cycles` is declared but
  also unused; its downgrade path is a separate sibling
  ticket (T8.d, not yet filed). Decision rationale: cycles
  are topological, not count-based, so the downgrade rule
  differs.
- **Coupling metric.** Same — `invalidates_coupling` has a
  different semantic (fan-in × fan-out aggregate) and lives
  in its own ticket.
- **Per-gap discriminator propagation.** The Desktop
  surface may eventually want to show *which* gap shape
  caused a particular downgrade. The
  `CoverageGap::discriminator()` method already exists
  (see `ports.rs`), but plumbing it through the report
  JSON is a separate wire-format decision.
