# Universal-fidelity sprint — follow-up registry

> **Purpose.** A single persistent home for concerns surfaced during sprint
> execution that do **not** block the current task but must not be lost.
> Every `§8 Out-of-scope follow-ups` entry in a task design doc that does
> not yet have a ticket should be mirrored here with a stable ID.
>
> **Do not edit closed rows.** When a follow-up is addressed, mark the
> `Status` column `Closed` and link the PR or commit. Leave the row in
> place so a future reader can audit what was filed vs what was resolved.
>
> **ID format.** `UF-FU-NNN` (universal-fidelity follow-up, zero-padded
> three-digit serial). IDs are assigned sequentially and never reused.

## How to file a new follow-up

1. Claim the next sequential `UF-FU-NNN` ID.
2. Fill every column. Empty cells force a reader to re-derive the
   context months later — do the work once, here, now.
3. If the follow-up is born from an `§8 Out-of-scope follow-ups` entry
   in a task design doc, cross-link both directions: the design doc
   cites the `UF-FU-NNN` ID, and the `UF-FU-NNN` row names the design
   doc in its `Source task` column.
4. Do not group similar items. Every distinct fix gets its own row so
   PRs can close them individually.
5. When the follow-up graduates to its own design doc (`docs/workstreams/universal-fidelity/tasks/<id>.md`),
   add a `Design doc` link to the row but keep the registry entry open
   until the implementation PR merges.

## UF-FU-021 — Rust extractor mis-classifies `pub use` re-exports as Function nodes

- **Opened**: 2026-04-18 (Tier 2.3, gridseak-self Round A audit, sample #3)
- **Owner**: universal-fidelity / proof-foundation-gap
- **Severity**: high (classifier-layer correctness; directly inflates
  `no_callers_high_confidence` independently of resolver state)
- **Summary**: the Rust tree-sitter extractor emits a `Function` node for
  `pub use digest::Update as DigestUpdate;` at
  `calibration/repos/jj-vcs__jj/lib/src/content_hash.rs:6`. `pub use` is a
  re-export of a trait (type), not a function — it has no body, no parameters,
  no return type, and cannot be called.  Phantom `Function` nodes always show
  zero outbound and zero inbound edges, so they land at the top of
  high-confidence `no_callers` every run.
- **Recipe**:
  - In the Rust symbol extractor, reject `use_declaration` tree-sitter nodes
    from the callable-set builder.
  - Emit `Import` or `Type`-re-export edges where the consuming graph stage
    expects them, so downstream cross-file resolution still works.
  - Add a regression fixture
    `graphengine-parsing/tests/rust_pub_use_not_function.rs` with a one-file
    Rust source containing a `pub use` and assert: (a) no `Function` node is
    emitted for it, (b) a corresponding `Import` edge is emitted if the
    re-exported symbol is in-corpus.
- **Exit**: `cargo test --test rust_pub_use_not_function` passes; gridseak-self
  rev *n+1* scan shows `DigestUpdate` absent from `no_callers_high_confidence`.
- **Notes**: orthogonal to UF-FU-012a (which is Layer-2 resolver behaviour).
  This bug inflates the numerator regardless of resolver fidelity.
